[1:00 — ARCHITECTURE]

(show assets/architecture_diagram.png)

Let me walk you through the architecture. Every user message first hits the security checkpoint node. This is a function node that scrubs PII like credit card numbers and phone numbers, detects prompt injection attempts, and enforces PIN verification for sensitive physical operations like unlocking doors.

If the message passes security, it routes to the orchestrator — an LLM agent that decides which specialist to call. For groceries and inventory questions, it delegates to the inventory manager agent. For repairs and scheduling, it delegates to the maintenance scheduler agent.

Both specialist agents connect to our local MCP server, which exposes five tools: get inventory, update inventory, get maintenance tasks, schedule maintenance, and verify owner PIN. The data persists in a local JSON file.

After the orchestrator gets results back, the workflow passes through a human approval node. This is our human-in-the-loop checkpoint — it pauses execution and asks the user to confirm any scheduling or inventory changes before they are finalized.

Finally, the approved result flows to the final output node, which formats the response for the playground UI.

[3:00 — SECURITY AND MCP]

The security checkpoint handles three things: PII scrubbing with regex for credit cards and phone numbers, prompt injection detection for keywords like "ignore instructions" and "system prompt", and a structured JSON audit log on every decision with severity levels from INFO to CRITICAL.

The MCP server runs locally via stdio transport, exposes five domain-specific tools, and is wired into both the inventory manager and maintenance scheduler agents.
